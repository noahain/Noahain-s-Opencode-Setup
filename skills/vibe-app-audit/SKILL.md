# Vibe-Coded App Audit

A fast, high-signal audit for apps generated by AI builders. These eight failures account for most of what breaks in production. Each one is verifiable in minutes — by reading the code or by clicking through the running app. Work through them in order; 1-3 and 8 are the ones that leak data or money, so never skip them.

## How to run this

1. Identify the stack first: frontend framework, backend/runtime, database, auth provider, and payment provider. Most vibe-coded apps use Supabase, Firebase, or a thin serverless backend plus Stripe.
2. For each check below, do the code review first, then confirm with the live test. Reading the code tells you *why* it breaks; the live test proves it.
3. Report findings as: severity, the exact file/line or request, what an attacker or unlucky user can do, and the smallest fix. Do not rewrite the whole app — patch the hole.
4. Re-test after each fix. A fix that isn't verified is a guess.

Severity guide: **Critical** = data or money exposed to anyone. **High** = breaks for real users or leaks under specific conditions. **Medium** = degrades experience or trust.

---

## 1. Broken object/data access (Critical)

**What breaks:** API responses return rows the logged-in user shouldn't see — other users' emails, IDs, or whole tables. The frontend hides the data; the backend hands it over.

**In the code:** Find every data-fetch endpoint or query. Look for queries with no `where user_id = :currentUser` clause, `select *` straight to the client, or Supabase/Firebase calls that rely on the client to filter. If using Supabase, check whether Row Level Security is actually enabled on each table (`alter table ... enable row level security`) and that policies exist — not just toggled on with no policy, which blocks nothing useful or everything.

**Live test:** Open dev tools → Network tab. Click through the app and read the JSON responses. If you see other users' records, identifiers, or fields the UI never shows, the table is readable.

**Fix:** Filter by the authenticated user server-side. Enable RLS with per-row policies. Return only the fields the client needs.

## 2. Missing ownership rules / IDOR (Critical)

**What breaks:** Login exists, but any logged-in user can read or edit anyone's records by changing an ID in the URL or request body.

**In the code:** For every route that takes a record ID (`/orders/:id`, `?doc=123`, `DELETE /item/:id`), confirm the handler checks that the record belongs to the current user before returning or mutating it. Authentication ("are you logged in") is not authorization ("is this yours").

**Live test:** Create users A and B. As A, open a URL containing A's data and copy it. Log in as B and paste it. If B sees or can edit A's data, ownership checks are missing. Repeat for write actions, not just reads.

**Fix:** On every record access, verify ownership (`where id = :id and user_id = :currentUser`) or an explicit shared-permission check. Return 404/403 on mismatch.

## 3. Payment lifecycle ignored (Critical)

**What breaks:** The build handles only the Stripe checkout-success path. Refunds, cancellations, chargebacks, and failed/expired payments are ignored, so users keep access after refunding and lose it after a transient failure.

**In the code:** Find the Stripe (or other) webhook handler. Check which events it handles. A correct integration reacts to at least `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed`, and `charge.refunded` / `charge.dispute.created`. Also confirm the webhook **verifies the signature** (`stripe.webhooks.constructEvent` with the signing secret) — an unverified webhook lets anyone grant themselves access by POSTing a fake event. Confirm access is driven by subscription state in your DB, not by a one-time flag set at checkout.

**Live test:** Trigger a refund (Stripe dashboard or `stripe trigger charge.refunded`). Confirm the user loses access. Trigger `invoice.payment_failed` and confirm the dunning/lockout behaves.

**Fix:** Verify webhook signatures. Map subscription lifecycle events to access state. Treat the DB subscription record as the source of truth, recomputed on every relevant event.

## 4. No error handling (High)

**What breaks:** A failed request leaves a white screen or an infinite spinner. Users on weak connections hit it constantly and churn without telling you.

**In the code:** Look for `fetch`/`axios`/mutation calls with no `catch`, no error state, no timeout. Look for async UI that only has loading and success states, never an error state. Forms that disable the button forever once clicked.

**Live test:** Throttle to "Offline" or "Slow 3G" in dev tools (or kill wifi) and submit a form / trigger a load. You should get a clear error and a way to retry — not a dead screen.

**Fix:** Wrap network calls in try/catch with user-visible error states and a retry. Add request timeouts. Add an error boundary so one failed component doesn't blank the page.

## 5. Leaked secrets (Critical)

**What breaks:** API keys, DB passwords, or service tokens pasted into AI chat logs during debugging, or committed to the repo, or shipped to the browser bundle.

**In the code:** Grep the repo and the client bundle for `key`, `secret`, `password`, `token`, `sk_live`, `sk_test`, `Bearer `, and provider-specific prefixes. Check that secrets live in server-side env vars, not in client code or `NEXT_PUBLIC_`/`VITE_` variables (those ship to the browser). Check git history (`git log -p`) — a deleted secret still lives in history. Confirm `.env` is gitignored.

**Manual check:** Search your AI builder's chat history for "key" and "password." Anything you pasted to debug is sitting in a transcript.

**Fix:** Rotate every exposed secret — assume it's compromised. Move secrets to server-side env vars. Scrub git history if needed (`git filter-repo`) and force-push. Never expose privileged keys to the client.

## 6. Mobile / low-end device breakage (High)

**What breaks:** Works on the builder's laptop, chokes on the mid-range Android most users actually have — layout overflow, tap targets too small, heavy bundles that time out.

**In the code:** Check for fixed pixel widths, missing viewport meta tag, hover-only interactions, and unoptimized large assets. Look at bundle size and whether images are sized/lazy-loaded.

**Live test:** Open the app in a mobile emulator (dev tools device toolbar) set to a mid-tier Android with CPU throttling, and on a real phone if possible. Walk the core flow.

**Fix:** Responsive layout, proper viewport meta, touch-friendly targets, compressed/lazy-loaded assets, and code-splitting to shrink the initial bundle.

## 7. Duplicate workflows / double-firing (Medium)

**What breaks:** Re-prompting the AI builder creates two copies of a flow — two onboarding paths, two webhook handlers, two cron jobs — so emails send twice and side effects double-fire.

**In the code:** Search for duplicated route handlers, duplicated event listeners, two email-sending code paths for the same trigger, or multiple scheduled jobs doing the same thing. Confirm side-effecting actions (emails, charges, writes) are **idempotent** — guarded by a key or a "already done" check so a retry can't repeat them.

**Live test:** Run a flow that sends email or writes a record and confirm exactly one happens. Trigger it twice quickly to test idempotency.

**Fix:** Delete duplicate paths. Add idempotency keys to side-effecting operations. Centralize each workflow in one place.

## 8. Missing role checks on admin (Critical)

**What breaks:** Admin routes and actions have no server-side role check — a logged-out visitor or a normal user can reach `/admin` or call admin APIs.

**In the code:** Find every admin route, page, and API. Confirm each enforces a role check **on the server** (`if (!user.isAdmin) return 403`), not just hidden navigation links. Hiding the button is not access control.

**Live test:** Type `/admin` (and other privileged URLs) while logged out, then as a normal user. Hit admin API endpoints directly with a normal user's session. Anything that loads or succeeds is a hole.

**Fix:** Server-side role enforcement on every privileged route and API. Default deny.

---

## Final report

Summarize as a pass/fail table across all eight, ordered by severity, with a one-line fix status for each. Call out anything Critical that's still open as a launch blocker. An app that passes all eight is ahead of ~90% of vibe-coded apps — say so, but only if it actually passes the live tests, not just the code read.
